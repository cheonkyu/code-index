# Item 32: 추상화 계약을 존중하라

## 추상화 계약이란 무엇일까요?

추상화 계약은 클래스나 인터페이스가 외부 세계에 어떤 기능을 제공하고, 어떤 조건을 만족해야 하는지를 정의하는 규칙이라고 생각하면 됩니다. 쉽게 말해, 특정 클래스를 사용하는 사람이 그 클래스에 대해 "이렇게 사용하면 예상대로 작동할 것이다"라는 믿음을 가질 수 있도록 하는 약속이죠.

이 계약은 크게 다음과 같은 요소들로 구성됩니다.

*   **Public API:** 클래스나 인터페이스의 `public`으로 선언된 함수, 프로퍼티, 생성자 등 외부에서 접근할 수 있는 부분
*   **Docstring (문서화):**  API의 동작 방식, 파라미터, 반환 값, 예외 등에 대한 설명
*   **클래스 불변식 (Class Invariants):**  클래스의 상태가 항상 만족해야 하는 조건 (예: 리스트의 크기는 항상 0 이상)

## 추상화 계약을 존중해야 하는 이유 (Best Practice)

추상화 계약을 존중하지 않으면 다음과 같은 문제가 발생할 수 있습니다.

*   **예상치 못한 동작:**  계약과 다른 방식으로 코드를 사용하면 예상하지 못한 오류가 발생하거나 잘못된 결과가 나올 수 있습니다.
*   **코드 유지보수의 어려움:**  추상화 계약이 명확하지 않거나 깨져 있다면, 코드를 수정하거나 확장할 때 예상치 못한 부작용이 발생할 위험이 커집니다.
*   **버전 업그레이드 문제:**  라이브러리나 프레임워크의 API가 변경될 때, 기존 코드가 제대로 작동하지 않을 수 있습니다.

따라서 추상화 계약을 존중하는 것은 코드의 신뢰성을 높이고, 개발 효율성을 향상시키는 데 필수적입니다.

## 어떻게 추상화 계약을 존중할 수 있을까요?

1.  **Public API만 사용하세요:**  클래스의 내부 구현에 직접 접근하지 말고, 항상 `public`으로 공개된 API를 통해서만 상호작용해야 합니다.  내부 구현은 언제든지 변경될 수 있으므로, 직접 접근하는 코드는 깨지기 쉽습니다.
2.  **문서화를 철저히 하세요:**  API의 동작 방식을 명확하게 설명하는 Docstring을 작성해야 합니다.  Docstring은 다른 개발자들이 코드를 이해하고 사용하는데 중요한 역할을 합니다.
3.  **클래스 불변식을 유지하세요:**  클래스의 상태가 항상 유효하도록 보장해야 합니다.  불변식을 깨뜨리는 코드는 버그를 유발할 수 있습니다.
4.  **API 변경 시 호환성을 고려하세요:**  API를 변경해야 할 경우, 기존 코드가 깨지지 않도록 호환성을 유지하는 방향으로 변경해야 합니다.  불가피하게 호환성이 깨지는 경우에는, 변경 사항을 명확하게 문서화하고, 이전 버전과의 호환성을 제공하는 방법을 고려해야 합니다.

## 예시

간단한 예를 통해 추상화 계약을 존중하는 방법을 보여드릴게요.

```kotlin
/**
 * 숫자를 저장하는 간단한 클래스
 *
 * @property value 숫자를 저장하는 프로퍼티
 */
class NumberHolder(val value: Int) {

    /**
     * 숫자를 2배로 반환하는 함수
     *
     * @return value의 2배
     */
    fun doubleValue(): Int {
        return value * 2
    }
}

fun main() {
    val holder = NumberHolder(5)
    val doubledValue = holder.doubleValue() // public API를 통해 접근

    println("두 배 값: $doubledValue")
}
```

위 예제에서 `NumberHolder` 클래스는 `value` 프로퍼티와 `doubleValue()` 함수를 제공합니다.  이 클래스를 사용하는 사람은 `value` 프로퍼티를 직접 변경하거나, 클래스 내부 구현에 접근하지 않고, `doubleValue()` 함수를 통해서만 숫자를 조작해야 합니다.  이렇게 함으로써 추상화 계약을 존중하고, 코드의 안정성을 확보할 수 있습니다.

## 핵심 요약

*   추상화 계약은 클래스나 인터페이스의 사용 방식을 정의하는 규칙입니다.
*   추상화 계약을 존중하면 코드의 신뢰성, 유지보수성, 호환성을 높일 수 있습니다.
*   `public` API만 사용하고, 문서화를 철저히 하며, 불변식을 유지해야 합니다.
*   API 변경 시에는 호환성을 고려해야 합니다.
